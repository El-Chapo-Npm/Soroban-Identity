# Issue #341: i18n Fallback to English Implementation Progress

**Branch Name:** `feature/i18n-fallback-to-english`
**Commit Message:** `feat: implement i18n fallback to English for missing keys - in-progress`

## Issue Summary
The i18n system doesn't configure a fallback locale. When a translation key exists in `en.json` but is absent from other locale files (like `es.json`), the translation function returns the raw key string instead of the English copy, showing internal key names as UI text.

## Planned Implementation

### 1. i18n Configuration Update
- Update `frontend/src/i18n.ts` to configure 'en' as fallback language
- Missing keys in non-English locales should silently resolve to English values
- Maintain backward compatibility with existing translations

### 2. Development Warning System
- Add console warning in development mode when fallback is used
- Warning format: `[i18n] Missing key '${key}' in locale '${lng}', falling back to 'en'.`
- Ensure warnings don't appear in production builds

### 3. CI Script for Missing Keys
- Create CI script that checks all non-English locale files for missing keys
- Script should compare each locale file against `en.json`
- Fail CI if any missing keys are found in non-English locales
- Script should be added to GitHub Actions workflow

### 4. Affected Files
- `frontend/src/i18n.ts` - Main i18n configuration
- `frontend/src/locales/` - All locale JSON files
- `.github/workflows/` - CI configuration for key checking

### 5. Testing Requirements
- Switching to locale with missing key renders English copy, not key string
- Development console shows expected warning for missing keys
- No warnings in production build
- CI script correctly identifies missing keys
- Existing translations remain intact
